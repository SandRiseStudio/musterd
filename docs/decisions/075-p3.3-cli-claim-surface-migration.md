# 075 — P3.3 CLI + MCP surface migration to the claim/agent-key handshake

- Status: proposed — 2026-06-29 (draft for team review; depends on P3.1/P3.2, gated by ADR 069)
- Date: 2026-06-29

## Context

ADR 069 phases the v0.3 governance rock; P3 is the breaking auth rework. Per ADR 069 decision 2 the
**shipped** end state is a **hard cutover** to *agent key + admin-issued grant* everywhere — no
localhost-keeps-token dual path. The build path (this ADR's reading of "gated dual-path") is to land
P3.1 (credential/grant substrate) and P3.2 (the `claim` WS handshake + request/approval lane) **behind
a gate** with the live `hello`/token auth still working, then **P3.3 removes the gate + the old path**
in one coordinated breaking moment across every surface.

The wire design is normative in `SPEC.md` Appendix A: A.0 (what changes), A.2 (credentials), A.3 (the
`claim` handshake replacing `hello`), A.5 (governance/request lane), A.7 (HTTP deltas), A.9 (surface
migration). This ADR is the **client-surface** half of P3.3 — the CLI (`@musterd/cli`) and the MCP
adapter (`@musterd/mcp`) — which is disjoint from the server/protocol lanes (P3.1/P3.2, Jasmine) and
the approval-UI web lane (Cleo/June, ADR 072).

As-built today (the surface P3.3 migrates):

- **Env:** `MUSTERD_TEAM` + `MUSTERD_MEMBER` + `MUSTERD_TOKEN` + `MUSTERD_SURFACE` (+ `MUSTERD_AUTOJOIN`,
  `MUSTERD_BINDING`), read by `packages/cli/src/config.ts` (`identityFromEnv`) and `packages/mcp/src/config.ts`.
- **Adapter env written by `init`:** `packages/cli/src/onboard/mcpEntry.ts` writes
  `MUSTERD_TEAM/MEMBER/TOKEN/SURFACE` into the harness (`claude mcp …` / `.cursor/mcp.json`).
- **`musterd claim`** (`commands/claim.ts`): the **local** claim-on-first-use (ADR 032) — mint-or-reuse
  via the unauthenticated `POST /members`, write `member+token` into `.musterd/binding.json`. Distinct
  from the A.3 `claim` WS frame: locally a seat is a member + its per-member token.
- **`musterd join`** (`commands/join.ts`): opens a WS `hello {team, as, token}` → presence.
- **`musterd team add`** (`commands/team.ts`): `POST /members` → prints a **token**.
- **`binding.json`:** `{ server, team, member, token, surface, claim? }` — the token is the credential.

## Problem

P3.3 must move every client surface from *token == member* to *agent key (authenticates a harness) +
claim (occupies a seat, with or without a pre-issued grant)*, atomically with the server cutover, while:

- keeping the **local frictionless onboarding** ADR 032/065 preserved in spirit — a fresh agent still
  goes from `init` → occupying a seat with near-zero steps; the difference is the credential class and
  the `claim` handshake, not the experience,
- handling the **pending** claim (no grant → admin approval) on the client: the WS stays open, the
  server pushes the terminal `occupied`/`refused` frame (spec-gap 3, no client polling), and a dropped
  socket re-`claim`s with server-side dedupe,
- migrating **env + binding** without a partial state that authenticates as the wrong seat, and
- building + testing the CLI surface **before** the server cutover lands, so P3.3 is a flip not a
  research project. That needs the P3.2 protocol frame schemas (`ClaimFrame`/`OccupiedFrame`/
  `RefusedFrame`/`PendingFrame`) to exist so the CLI can be unit-tested against a mock claim server.

## Decision

### 1. New credential env + binding shape (replaces token)

| Today | P3.3 |
|---|---|
| `MUSTERD_TOKEN` | `MUSTERD_AGENT_KEY` (the harness-authenticating team key) |
| `MUSTERD_MEMBER` | `MUSTERD_CLAIM` — `{seat:<name>}` | `{role:<name>}` | `{observe:true}` |
| — | `MUSTERD_GRANT` (optional pre-issued grant token; omit → request lane) |

`MUSTERD_TEAM` / `MUSTERD_SURFACE` / `MUSTERD_BINDING` / `MUSTERD_AUTOJOIN` are unchanged. The
`binding.json` shape gains `agent_key` + `claim` + optional `grant` and **drops `member` + `token`**;
the resolved seat name is learned from the `occupied` frame (not assumed client-side), so a role-pool
claim binds to the concrete `<role>-<n>` the server assigned. `MUSTERD_AGENT_KEY`/`MUSTERD_GRANT` are
secrets — same handling as the token today (chmod-600 binding, never logged, never in `--json`).

### 2. `musterd claim` / `musterd join` → the `claim` handshake

Both open the WS `claim` frame (A.3): `{ key, target, grant?, surface }` → `occupied | refused | pending`.

- `occupied` → write the resolved seat + agent key (+ grant if any) into `binding.json`; print
  `✓ <seat> — occupied on <team>` (+ `charter` if served). Exit 0.
- `refused` → map `code` to the exit-code table: `claim_conflict`→9, `forbidden`→5, `not_found`→6,
  `disabled`/`banned`→5 (with the A.8 refusal copy), `expired_grant`→5. Print the `claimable` + `hint`.
- `pending` → print `asked admins to authorize this claim (request <id>)` and **hold the socket open**
  until the server pushes the terminal `occupied`/`refused` frame (spec-gap 3). `--timeout` bounds the
  wait (mirror `inbox --wait`'s 124-on-timeout). A dropped socket re-`claim`s; the server dedupes by
  `(team, from_session, target)`.

A **stateless mirror** `POST /teams/:slug/claim` (A.7) backs a one-shot CLI path for harnesses that
can't hold a WS (the `--no-wait` form: submit, print `pending` + `request_id`, exit 0; the approval
lands as a normal inbox act the agent picks up later).

### 3. `musterd team add` → seat, no token

`POST /teams/:slug/seats` (A.7) provisions a seat in `provisioned` and returns the **seat** (no token).
Output prints the seat + the **claim target** + the team's `agent_key` (once, for the operator to hand
the agent) — the ready-to-paste connect hint changes from "set `MUSTERD_TOKEN`" to "set
`MUSTERD_AGENT_KEY` + `MUSTERD_CLAIM` (+ optional `MUSTERD_GRANT`)". A pre-issued grant is created via
the new `musterd grant` command (P3.1 admin HTTP) when the admin wants the no-prompt path.

### 4. `musterd init` + the adapter env

`init` writes the new env triple into the harness (`mcpEntry.ts`): `MUSTERD_AGENT_KEY` + `MUSTERD_CLAIM`
+ `MUSTERD_GRANT?` (instead of `MUSTERD_MEMBER`/`MUSTERD_TOKEN`). The MCP adapter
(`packages/mcp/src/config.ts` + `bind.ts`) reads the new triple; `team_join` opens the `claim`
handshake (instead of `hello`), and `MUSTERD_AUTOJOIN=1` auto-claims on launch. The pending-marker
resolution path (ADR 033/034) is preserved — a pending claim resolves via the same `claim` frame.

### 5. Build behind a gate; P3.3 is the flip + removal

P3.1/P3.2 land with the live `hello`/token auth still working (gated dual-path during dev, behind e.g.
`MUSTERD_AUTH=claim` or a daemon config flag). The CLI claim/join code is built + unit-tested against a
mock claim server using the P3.2 protocol frame schemas, **before** the server cutover. P3.3 then:
(a) flips the default to claim/key, (b) removes the `hello`/token path from the CLI + MCP + server, (c)
renames the env in `init`/`mcpEntry`, (d) documents the one-shot reset (decision 1) — existing
`member+token` bindings are stale after the cutover; `musterd init` re-issues an agent key + claim.

### 6. Lane + sequencing

- **P3.2 protocol frame schemas** (`ClaimFrame`/`OccupiedFrame`/`RefusedFrame`/`PendingFrame`,
  `RefusedCode`) are the prerequisite — the CLI imports them. Jasmine's lane (protocol); this ADR
  depends on them but does not define them (deferred to the P3.2 ADR).
- **P3.3 CLI + MCP** (this ADR): `claim.ts`/`join.ts` rewrite, `config.ts` env, `mcpEntry.ts`,
  `packages/mcp` config/bind, `team add` output. Mine.
- **Server cutover + `hello` removal**: Jasmine's lane; coordinated in the P3.3 landing commit set.

## Consequences

- One coordinated breaking change across `init` / `team add` / MCP env / `join` / `claim` (decision 2);
  no localhost-keeps-token path ships. The gated dual-path is dev-only and removed at cutover.
- The local frictionless experience is preserved: `init` still gets a fresh agent to an occupied seat in
  near-zero steps; the credential is an agent key + claim, not a member token. Role-pool claims
  (`MUSTERD_CLAIM=role:backend`) keep the "offer free seats" recipe (ADR 055 no-dead-end) intact.
- Existing `binding.json` files (`member+token`) are stale after cutover — the one-shot reset (decision
  1) re-issues credentials; `init` re-binds. Documented, not auto-migrated.
- The pending-claim wait is a new resident client state (a held WS), mirroring `inbox --watch`/`--wait`;
  the no-WS harnesses get the stateless `POST /claim` + later inbox pickup.
- Builds on ADR 069 (the plan + decision 2), ADR 032/033/034 (local claim + pending + live delivery —
  preserved in spirit), ADR 055 (no-dead-end conflict hints), ADR 058 (binding file), ADR 018 (identity
  resolution ladder), SPEC A.3/A.7/A.9.

## Observability & Evaluation

**Traces** — the `claim` handshake is itself a governed, audited event: every `occupied`/`refused`/
`pending`→`occupied` writes an audit row (P3.1 extends ADR 071's verbs with `claim.occupy`,
`claim.refused`, `grant.use`). The CLI emits a `status_update` on a successful occupy (so the roster
reflects the new seat) and a `request_help` is **not** auto-sent on `pending` — the request lane is the
server's admin notification, not a team broadcast. A `--debug` path logs the frame sequence
(`claim→pending→occupied`) to stderr behind `MUSTERD_DEBUG=1`.

**Eval** — success = "the cutover swaps auth without breaking the onboarding loop". The paired
regression guard: (1) a fresh `init` → claim → `occupied` round-trip binds the seat and the agent can
`send`/`inbox`; (2) a no-grant claim → `pending` → admin approve (local fast path) → `occupied`;
(3) a `claim_conflict` on an occupied agent seat → exit 9 with the `claimable` + `hint`; (4) a dropped
pending socket re-`claim`s and dedupes to the same request; (5) `--json` emits the `occupied`/`refused`
shape with no ANSI; (6) the stateless `POST /claim` mirror returns `pending` + `request_id`. Baseline:
the current 542-test green suite (the token-based claim/join paths this replaces) — each new claim path
ships with the paired test that is its regression guard, and the cutover commit keeps the suite green
by moving every surface in one change.

**Experiment** — none yet. A future one (per ADR 069's batond experiment): does the claim/grant model
raise time-to-first-`occupy` or admin-interrupt count vs the v0.2 token model on seeded onboarding +
reconnect tasks, and does the TTL-grant lifetime cut reconnect re-prompts to ~zero without widening the
exposure window? The audit log is the measurement substrate.
