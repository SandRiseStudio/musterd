# 078 — P3.2 protocol frame schemas for the `claim` handshake

- Status: accepted — 2026-06-29 (P3 re-carve: Riley takes the protocol frames per June's handoff + Nick's gate)
- Date: 2026-06-29

## Context

P3 (ADR 069) is the breaking auth rework. Nick's directional call (2026-06-29): assume Jasmine
unavailable, take over her lanes; hard cutover, no dev gate (ADR 075). June's 17:34 handoff re-carved
P3 into parallel lanes — **riley → P3.2 protocol frames** (land first, the shared dependency), **June
→ P3.1 substrate** (ADR 076), **Cleo → P3.2 WS claim handshake + request lane** (ADR 077) — converging
on one atomic cutover. The frames are the prerequisite: June's grant/credential storage + Cleo's WS
handshake + my P3.3 CLI (ADR 075) all import them.

The wire design is normative in `SPEC.md` Appendix A: A.3 (the `claim` handshake), A.2 (credentials),
A.5 (governance/request lane), A.8 (refusal/error codes). This ADR adds the frame **schemas** to
`@musterd/protocol` — the executable contract — so the three lanes build against one shape. It does
**not** wire the frames into the live WS path (that is Cleo's P3.2 cutover step, part of the atomic
merge) and does **not** remove `hello` (that is the cutover).

As-built today: `frames.ts` carries the live `hello`/`welcome`/`subscribe`/`send`/`heartbeat` +
server frames; `errors.ts` carries 10 `ErrorCode`s; `audit.ts` carries `AuditEntry`/`AuditResponse`
with `action` an open string (ADR 074). No `claim`/`occupied`/`refused`/`pending` frames; no
`claim_conflict`/`expired_grant` codes.

## Problem

Three lanes need a stable wire contract before they can build in parallel, but the frame shapes carry
real design decisions (the `RefusedCode` membership, the `target` shape, the `seat` representation, the
`v` pin, the `expired_grant` HTTP status). Drafting them risks pre-committing Jasmine's design. We
mitigate by drafting **from the normative SPEC** (A.3 field shapes are specified, A.8 lists the refusal
codes) and surfacing every design call in this ADR for team review — so the risk is "Jasmine adjusts a
field on her return," not "three lanes rebuild." Landing the schemas first (vs. each lane defining its
own) is what makes the parallel build non-overlapping.

## Decision

### 1. New `claim-handshake.ts` module (SPEC A.3)

- `ClaimTargetSchema` = `z.union([{seat:string},{role:string},{observe:z.literal(true)}])`. A plain
  union (the variants carry different keys; no single discriminator).
- `ClaimFrame` — `{ type:'claim', v:PROTOCOL_VERSION, team, key:string, target, grant?:string, surface }`.
  `key` = agent key or human credential (secret, hashed at rest, never logged). `grant` present →
  occupy; omitted → open a claim request (A.5).
- `OccupiedFrame` — `{ type:'occupied', seat:MemberSchema, presence_id, server_time:int, charter?:string,
memory:null }`. `memory` is a reserved seam, **always null in v0.3** (enforced `z.null()`).
- `RefusedFrame` — `{ type:'refused', code:RefusedCode, message, claimable:string[], hint:string }`.
- `PendingFrame` — `{ type:'pending', request_id, message }`.
- `RefusedCodeSchema` = `claim_conflict|forbidden|not_found|disabled|banned|expired_grant` (A.8).

### 2. `ErrorCode` extended (SPEC A.8)

Add `claim_conflict` (HTTP 409 — seat occupied) and `expired_grant` (HTTP 410 — grant past lifetime) to
the shared `ErrorCode` + `ERROR_HTTP_STATUS`. `disabled`/`banned` are **WS-only** refusal codes (HTTP
maps a disabled/banned seat's credential rejection to `forbidden` 403 per A.2) — they live in
`RefusedCodeSchema`, not `ErrorCode`.

### 3. `P3_AUDIT_ACTIONS` reference vocabulary

A `const` tuple naming the P3 audit verbs (`grant.issue/use/revoke`, `claim.occupy/refused`,
`request.decide`, `key.rotate`, `policy.change`, `account_status.change`) so June's substrate + Cleo's
handshake emit one consistent vocabulary. `AuditEntry.action` **stays an open string** (ADR 074) — the
tuple is reference-only, not an enum; a verb outside it still validates.

### 4. Not wired into the live WS unions

`ClaimFrame` is **not** added to `WSClientFrame`; the server frames are **not** added to
`WSServerFrame`. Wiring + `hello` removal is Cleo's P3.2 cutover step, landing in the atomic merge.
This keeps the schema commit purely additive + non-breaking — `hello` still works, the live daemon is
untouched, and the team builds against the new shapes on their branches.

## Design calls surfaced for review

- **`v` pin** — `ClaimFrame.v` pins to the current `PROTOCOL_VERSION` (`musterd/0.3`). Additive frame
  types need no version bump to ship the schema; the breaking handshake swap is the cutover's job. If
  the cutover wants a handshake version bump, that lands in the P3.3 cutover ADR set, not here.
- **`seat` on `occupied`** — uses the existing `MemberSchema` (P1 already carries `account_status` +
  `capabilities`). A dedicated `SeatSchema` with `occupied_by`/`activity`/`charter` is reserved for
  when those fields finalize; `charter` rides top-level on `occupied` per A.3.
- **`expired_grant` HTTP status** — A.8 says "410/403"; this ADR picks **410 (Gone)** as the precise
  semantic. If the team prefers 403, it's a one-line change in `ERROR_HTTP_STATUS`.
- **`RefusedCode` membership** — exactly A.8's list (`claim_conflict`, `forbidden`, `not_found`,
  `disabled`, `banned`, `expired_grant`); `archived` seats refuse as `not_found` (an archived seat is
  effectively gone), not a new code.

## Consequences

- June's P3.1 (ADR 076) + Cleo's P3.2 (ADR 077) + my P3.3 (ADR 075) import `ClaimFrame`/
  `OccupiedFrame`/`RefusedFrame`/`PendingFrame`/`RefusedCode` from `@musterd/protocol` — one contract,
  no triple-definition.
- The schemas land **first** (this ADR) so the parallel build is non-overlapping; they are additive and
  non-breaking (`hello` still works, frames not yet in the WS unions).
- `ErrorCode` gains two codes — the CLI (`04-cli.md` exit-code map) + the server error renderer must
  learn `claim_conflict` (→ exit 9, reusing the existing `conflict`/`member_busy` 409 mapping) and
  `expired_grant` (→ exit 5). That CLI mapping lands in the P3.3 cutover, not here.
- Builds on ADR 069 (the plan + decision 2), ADR 075 (the hard-cutover CLI surface), ADR 074 (open
  audit `action`), ADR 071 (audit log), SPEC A.2/A.3/A.5/A.8.

## Observability & Evaluation

**Traces** — the schemas carry no runtime behavior themselves (no server wiring yet). The
observability they _enable_: the `claim` handshake is a governed, audited event — every `occupied`/
`refused` writes an audit row named from `P3_AUDIT_ACTIONS` (`claim.occupy`, `claim.refused`), and
every grant issue/use/revoke writes `grant.*`. Those verbs flow to the `GET /teams/:slug/audit` log
(ADR 071) + the web audit view (ADR 073). The frame schemas are the shape the audit `detail` payloads
ride on.

**Eval** — success = "the three P3 lanes import a stable frame contract and the schema commit is
non-breaking." Guards (in `claim-handshake.test.ts`): (1) each frame round-trips a SPEC A.3 example;
(2) the three `ClaimTarget` variants parse, an empty target rejects; (3) `occupied.memory` rejects
non-null (the reserved seam stays unwired); (4) every `RefusedCode` parses, an unknown one (`archived`,
`ok`) rejects; (5) `ClaimFrame.v` rejects a wrong version pin; (6) `ErrorCode` accepts the two new
codes; (7) `P3_AUDIT_ACTIONS` contains the P3 verbs while `action` stays open. Baseline: the 77-test
protocol suite (was 64) stays green — the new frames are additive, no existing test moves.

**Experiment** — none yet. The handshake's governance overhead vs the v0.2 token model is ADR 069's
batond experiment (calibrated once the cutover lands + the audit log is the measurement substrate).
