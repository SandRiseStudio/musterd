# 071 ŌĆö v0.3 P2: in-band enforcement + audit (on the existing token auth)

- Status: accepted ŌĆö 2026-06-29 (P2 of ADR 069)
- Date: 2026-06-29

## Context

ADR 069 phased the v0.3 governance rock; ADR 070 (P1) landed the **substrate** ŌĆö every seat now carries
resolved `account_status` + effective `Capabilities` (Universe-1 enforced fields + Universe-2 declared
fields), projected by `reconcile` from the git seat-files with the narrowing-only rule. P1 shipped
**permissive defaults and zero enforcement**.

**P2 turns the substrate on.** It is the first real governance value and, per ADR 069, it **completes the
Wave-2 notification-tiers item** (the `urgent` breakthrough is finally gated on a capability rather than
ungated). Critically, P2 ships on the **existing token auth** ŌĆö occupant == seat, `authMember` is
unchanged ŌĆö so there is **no flag day**. The breaking auth rework (agent key + claim/grant) is isolated to
P3.

As-built before P2 (the seams ADR 069 catalogued, each carrying a `// the v0.3 seat model will gate this`
marker): `meta.urgent` is schema-validated but **ungated** (`envelope.ts`); `reclaim`/`remove` are
**ungated** operator escape hatches (`http.ts`, any member may call them); the roster projection
(`summarize` ŌåÆ `toMember`) is **un-scoped** (every viewer sees every field); observer-mute is the one
enforced proto-capability (`route.ts`). There is **no audit log**.

## Problem

P2 must add real authorization at five seams + an audit trail **without** breaking an existing local
session that has no roles, no admin, and no per-seat narrowing (the generalist default = "everyone may do
everything"). Two of the seams have backward-compat traps:

1. **The `is_admin` gate on `reclaim`/`remove`.** The generalist default is `is_admin:false`, and P1 never
   implemented the "creator seat defaults admin" note. So a naive gate would lock **everyone** out of the
   operator escape hatch on every existing team ŌĆö a regression, not enforcement.
2. **The `urgent` gate.** Rejecting an over-reaching `urgent` send outright would surface as a hard error on
   a previously-working path. The notification-tiers design wants the message to still arrive ŌĆö just not as
   a breakthrough.

We also need a normative audit schema (the cross-cutting trace artifact ADR 069 introduces and the batond
flywheel consumes) before the gates can write to it.

## Decision

### 1. Audit log ŌĆö the append-only governance trace (foundational)

A new `audit` table (migration **v9**, additive ŌĆö no reset needed; existing teams gain an empty log):

```
audit(id, team_id ŌåÆ teams(id) ON DELETE CASCADE, ts, actor, action, target, result, detail, created_at)
```

- `actor` ŌĆö the member name that initiated the op (nullable: system/reaper writes have no actor).
- `action` ŌĆö dotted verb: `urgent.flagged`, `urgent.denied`, `send.denied`, `member.reclaim`,
  `member.remove`, `observe.denied`. (P3 adds `grant.*`, `claim.*`, `account_status.change`, `key.rotate`,
  `policy.change`, `request.decide` ŌĆö the schema is already shaped for them.)
- `result` ŌĆö `allow` | `deny` (the authorization outcome; an executed governance op is `allow`).
- `target` ŌĆö the affected seat/resource name (nullable).
- `detail` ŌĆö JSON blob for context (`{ reason }`, `{ fallback: "no-admin" }`, ŌĆ”); never carries secrets.

`store/audit.ts`: `appendAudit(db, teamId, entry)` (single INSERT, never throws into the request path ŌĆö
auditing is best-effort observability, not a gate) + `listAudit(db, teamId, { limit, before })`
(newest-first, capped). Append-only: no update/delete API. **`GET /teams/:slug/audit` is admin-only**
(`visibility_level === 'admin'`, i.e. `is_admin`), since the log exposes who-did-what across the team.

### 2. `can_flag_urgent` ŌĆö downgrade-and-deliver, never reject (completes notification tiers)

In `routeEnvelope`, after identity/observer checks, resolve the sender's effective capabilities. If
`meta.urgent === true`:

- **Allowed** (`can_flag_urgent`): deliver as urgent; `appendAudit('urgent.flagged', allow, { reason })`
  (the scarce flag is auditable, per the ADR 044 comment).
- **Denied** (`!can_flag_urgent`): **downgrade and still deliver** ŌĆö strip `meta.urgent`, set
  `meta.wasnt_urgent: true` (so the recipient and the firehose see "this seat tried to break through but
  isn't allowed to"), keep `urgent_reason` for context; `appendAudit('urgent.denied', deny, { reason })`.
  The message lands as an ordinary act, never a hard error on a previously-working path.

The downgrade mutates the envelope **before** `insertMessage`, so the persisted log and every delivery
(direct, firehose) carry the corrected meta.

### 3. `account_status` + `can_message` send gates

In `routeEnvelope`, before resolving recipients:

- `resolveAccountStatus(sender)` Ōłł {`disabled`, `banned`, `archived`} ŌåÆ `forbidden` (`send.denied`,
  `{ account_status }`). `provisioned`/`active` send normally.
- `can_message === 'none'` (muted) ŌåÆ `forbidden` (`send.denied`, `{ can_message: 'none' }`).

Both audit the denial. Generalist defaults (`active`/derived, `can_message:'team'`) pass untouched ŌĆö no
regression.

### 4. `is_admin` gate on `reclaim`/`remove` ŌĆö creator-admin default + empty-admin fallback

Three coordinated moves so enforcement is real on new teams without breaking existing ones:

- **Creator-admin default.** `POST /teams` mints the creator's human seat with
  `capabilities = { ŌĆ”generalist, is_admin: true }` persisted (the ADR 070 stated default, implemented
  here). New teams have a real admin from birth.
- **The gate.** `reclaim`/`remove` require the caller's effective `is_admin`.
- **Empty-admin fallback (the no-flag-day guarantee).** If the team has **zero** seats with `is_admin`
  (a legacy/un-migrated team, or a db-only team created before this change), the gate falls back to the
  v0.2 behaviour (any member may call), and the audit entry records `{ fallback: "no-admin" }`. The moment
  any seat gains admin (creator default, or a seat-file `[capabilities] is_admin = true`), the fallback
  closes and enforcement is live. This is the "value before the breaking change, no flag day" contract of
  ADR 069 made concrete.

Both ops write `member.reclaim` / `member.remove` audit entries (with the fallback marker when it applied).
`unbind` stays self-authed and ungoverned (you may always release your own seat).

### 5. `visibility_level` ŌĆö viewer-scoped roster projection

`summarize` becomes viewer-aware. The roster reads (`GET /teams/:slug`, `GET /teams/:slug/members`) stay
**unauthenticated-friendly** (no auth requirement added ŌĆö that would be a breaking change), but when a valid
bearer token is present the viewer is resolved:

- **admin viewer** (`is_admin`) ŌåÆ the full projection (every seat's effective `capabilities` +
  `account_status`).
- **team viewer / unauthenticated** ŌåÆ the **need-to-know** projection: a seat sees its **own** full
  capabilities, but other seats' `capabilities` are omitted (handles, kind, role, presence, availability,
  and account_status remain ŌĆö the coordination-visible surface). This hides the authority map (who is admin,
  who is muted, who is narrowed) from non-admins, matching SPEC A.1's `team` level.

The scoping is layered in the HTTP `summarize` step **on top of** `toMember`, so P1's reconcile/`toMember`
golden test (which asserts the full projected caps) is unchanged; P2 adds new viewer-scoped assertions at
the HTTP roster layer instead.

### 6. `can_observe` ŌĆö formalize the observer capability

An observer seat watches the firehose (ADR 061/063). P2 gates the firehose subscription on the subscriber
seat's `can_observe`: a seat narrowed to `can_observe:false` is refused the `team-all` subscription
(`observe.denied`). Generalist `can_observe:true` keeps every observer working.

### What P2 does NOT do

No credentials, no agent key, no claim/grant handshake, no request/approval lane, no `account_status`
**write** API (admins set it in the seat-file today; the HTTP write endpoint is P3) ŌĆö all P3. No schedule
enforcement / urgent rate-limiting (Wave 3). Auth is still occupant-token == seat.

## Observability & Evaluation

**Traces** ŌĆö P2 _introduces_ the audit log: every governed decision (`urgent.flagged/denied`,
`send.denied`, `member.reclaim/remove`, `observe.denied`) appends `{ ts, actor, action, target, result,
detail }`. This is the coordination-governance trace no single-agent observability tool can produce and a
direct feed for the batond flywheel (ADR 051). The existing `withEnvelopeSpan` is unchanged; a denied/
downgraded send still spans (with the corrected meta), so the trace and the audit row agree.

**Eval** ŌĆö success = **enforcement without regression**. The bar: the full suite stays green; the
generalist-default team behaves exactly as v0.2 (urgent works, any member reclaims via the empty-admin
fallback, every seat sees the roster). New assertions per seam: a `can_flag_urgent:false` seat's urgent is
delivered-but-downgraded (not rejected) and audited; a `disabled`/`banned` seat's send is forbidden + audited;
a non-admin is refused `reclaim` **once an admin exists** but allowed under the empty-admin fallback; a
non-admin roster read omits other seats' caps; `GET /audit` is admin-only. **Baseline/dataset**: the v8
roster output + the green suite are the "before"; each new gate ships with the paired allow/deny test that is
its regression guard.

**Experiment** ŌĆö named for batond (ADR 069): A/B the governance posture ŌĆö does the downgrade-and-deliver
urgent policy (vs hard-reject) measurably reduce sender retries / confusion on seeded notification-tier
tasks, and does the empty-admin fallback ever fire in real onboarding (a signal that the creator-admin
default isn't reaching file-backed teams)? The audit log is the measurement substrate.

## Consequences

- The highest-value governance (capability enforcement + audit, which finishes notification tiers) lands on
  the existing token auth ŌĆö value before P3's breaking change, exactly as ADR 069 sequenced.
- The audit table is shaped now for the P3 verbs (`grant.*`, `claim.*`, `account_status.change`, ŌĆ”) so P3
  adds rows, not schema.
- The empty-admin fallback bounds the blast radius of turning enforcement on: no existing team loses its
  escape hatch, and enforcement self-activates the instant an admin is declared. The trade-off is a window
  where an adminless team is ungoverned ŌĆö acceptable for a pre-1.0 local product, and visible in the audit
  (`fallback: "no-admin"`).
- The viewer-scoped projection is the first behaviour change to the roster since P1; consumers that relied
  on seeing every seat's caps (only the admin dashboard should) move behind an authenticated admin read.
- Builds on ADR 070 (the substrate), ADR 069 (the plan + the four decisions), ADR 044 (the urgent mechanism
  this gates), ADR 063 (the observer this generalizes), ADR 058 (the seat-files that carry the durable
  admin-set fields), ADR 026 (Universe-1 vs Universe-2 ŌĆö only Universe-1 is gated here).
