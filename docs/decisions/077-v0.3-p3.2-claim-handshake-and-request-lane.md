# 077 — v0.3 P3.2: WS claim handshake + request/approval lane

- Status: accepted — 2026-06-29 (P3.2 of ADR 069, re-carved from Jasmine's original lane)
- Date: 2026-06-29

## Context

ADR 069 phased the v0.3 governance rock. P3.1 (ADR 076, June) lands the credential/grant substrate:
`grants` + `requests` tables, migration v10, `agent_key_hash` on teams, minting helpers
(`mskey_`/`msgr_`/`mscr_` prefixes), and the admin HTTP endpoints for key rotation + policy. **P3.2
(this ADR) layers the WS claim handshake on top of that substrate** — the server-side behaviour change
that makes `claim` replace `hello` and drives the request/approval lane.

Per Nick's P3 decisions (ADR 069 + the re-carve): **immediate cutover, no dev gate** — `claim/grant`
replaces `hello/token` in the same atomic merge; no dual-path is shipped. This ADR builds on a feature
branch against in-memory daemons. The live dogfood daemon resets once during the coordinated P3 cutover
(ADR 069 decision 1). **Security-sensitive** — the credential auth path and the admin approval lane
carry a security-review gate before the atomic P3 merge.

## Problem

The `hello` frame authenticates a session by member token (one secret = one identity). After P3, a
session connects unauthenticated and must `claim` a seat: it proves *harness identity* (agent key, hashed
team-scoped secret) and *authorization* (an admin-issued grant). The server must handle three outcomes:

1. **Occupied** — key valid, grant valid, seat free → session becomes the seat occupant.
2. **Refused** — key valid but seat taken, grant expired, account disabled/banned, etc.
3. **Pending** — key valid but no grant → server creates a request, notifies admins, holds the WS
   open and **pushes** the terminal frame (`occupied` or `refused`) when the admin decides. No client
   polling (spec-gap 3, ADR 069).

The server also needs the admin-side request lane: `GET /requests` + `POST /requests/:id/decide`, where
`decide` issues a grant and pushes `occupied` to the waiting claim (or `refused` on deny).

## Decision

### Wire contract (SPEC A.3 / A.8 — normative)

**Client → Server `claim` frame:**
```
{ type:"claim", v:<version>, team:<slug>, key:<agent-key | human-credential>,
  target: {seat:<name>} | {role:<name>} | {observe:true},
  grant?:<grant-token>,   // omitted → no-grant path → pending
  surface:<string> }
```

**Server → Client outcomes:**
- `occupied` — `{ type:"occupied", seat:<Seat>, presence_id, server_time, charter?, memory:null }`
- `refused` — `{ type:"refused", code:<RefusedCode>, message, claimable?, hint? }`
- `pending` — `{ type:"pending", request_id, message }`

**RefusedCode** (A.8): `claim_conflict` | `forbidden` | `not_found` | `disabled` | `banned` |
`expired_grant`.

### Claim flow (server-side)

1. Parse `ClaimFrame`; reject unknown version.
2. Resolve team by slug.
3. Authenticate the key: `sha256(frame.key)` against `teams.agent_key_hash` (agent key) **or** the
   target seat's `credential_hash` (human credential). → `refused {forbidden}` on mismatch.
4. Resolve target seat: by `frame.target.seat` name or next open seat in `frame.target.role` pool.
   → `refused {not_found}` if no seat exists and role pool is full.
5. Check `account_status`: `disabled`/`banned`/`archived` → `refused {disabled|banned}`.
6. Check single-active: if seat is occupied → `refused {claim_conflict, claimable:[…], hint}`.
7. **Grant path** (grant token present): validate `sha256(frame.grant)` against `grants` table —
   not expired, not revoked, scoped to this seat/role. → on valid: OCCUPY (step 9).
   → on invalid/expired: `refused {expired_grant}`.
8. **No-grant path**: create `requests` row (`pending`), notify admins (route to connected admin seats
   via `hub.deliverToAdmins()`), send `pending {request_id}` to client. Hold WS open (spec-gap 3 —
   server pushes terminal frame when admin decides, no client polling). Deduplication: if a `pending`
   request already exists for `(team, session_conn_id, target)`, reuse it (idempotent, no duplicate
   admin prompts). Start/extend the 1h expiry window (spec-gap 2).
9. **OCCUPY**: insert/update occupancy on the seat (replaces the `hello` path's `authMember` +
   `upsertPresence`). Emit `presence` event to the team. Send `occupied` frame to client.
   Mark `awaitingClaim: false` on the hub connection.

### `POST /requests/:id/decide`

Admin-only (`authAdmin`, strict `is_admin`). Body:
```
{ approve: boolean, lifetime?: "once"|"ttl"|"standing", ttl_hours?: number,
  create_seat?: { name?: string, role: string } }
```

On `approve`:
1. If `create_seat` present: mint new seat in the role, add to seat-file + reconcile.
2. Issue a grant (per `lifetime`, defaults to `once`): insert into `grants` table.
3. Push `occupied` frame to the waiting WS connection (`hub.deliverClaimDecision(connId, occupied)`).
4. Mark request `approved`, `decided_by`.
5. Audit: `request.decide {result:allow, lifetime}`.

On `deny`:
1. Push `refused {forbidden}` to the waiting WS connection.
2. Mark request `denied`, `decided_by`.
3. Audit: `request.decide {result:deny}`.

Both outcomes: if the waiting connection has already disconnected, mark the request accordingly and
return 200 (decision still recorded).

### `GET /requests`

Admin-only. Returns pending (and optionally settled) requests for the team, newest-first. Query params:
`?status=pending|approved|denied|expired&limit=…&before=…`.

### Request expiry reaper (spec-gap 2)

Extend the existing presence/grace reaper in `server/src/store/presence.ts` (or a new
`store/requests.ts` sweep): mark requests `expired` where `ts + 3_600_000 < now` and status is still
`pending`. Push `refused {expired_grant}` to any still-open waiting WS connection before closing the
pending state. Sweep interval: same cadence as the grace reaper (~5s).

### Hub additions

- `Connection.awaitingClaim?: string` — the `request_id` while in pending state (cleared on terminal
  frame delivery).
- `hub.deliverClaimDecision(connId, frame)` — push `occupied`|`refused` to a specific connection by
  its conn id; used by `/decide` and the expiry reaper.
- `hub.deliverToAdmins(teamSlug, frame)` — broadcast a governance notification to all connected seats
  with `is_admin` on the team; used to surface pending requests to co-present admins.

### Approval UI wiring (web)

Once P3.1's `GET /requests` + `POST /requests/:id/decide` endpoints are live, wire:
- `packages/web/src/live/client.ts` — add `fetchRequests(team, token)` + `decideRequest(...)`.
- `packages/web/src/routes/approvals.tsx` — swap June's fixture array for live `fetchRequests`.
- `ApprovalCard.onApprove` / `onDeny` → `decideRequest(...)`.

This is the final piece of the approval-UI lane and ships in the same atomic P3 merge.

### What P3.2 does NOT include

No changes to the `hello`/token auth path during build — that remains for the dogfood team until the
P3.3 atomic cutover. `claim` is built alongside `hello`; P3.3 removes `hello` + token auth in the same
cutover merge. No CLI surface changes (riley, ADR 075). No `team add` or `init` migration (ADR 075 /
P3.3). No cross-network join (P4).

## Observability & Evaluation

**Traces** — extend the audit log (ADR 071 substrate):
- `claim.occupied {seat, surface, grant_lifetime?}` — allow
- `claim.refused {code, target}` — deny
- `claim.pending {request_id, target}` — allow (created)
- `request.decide {approve, lifetime?, decided_by}` — allow/deny
- `request.expired {request_id}` — system (no actor)

**Eval** — success = the three-path claim flow (occupied/refused/pending) passes in integration tests
against an in-memory daemon with the P3.1 substrate; the no-grant → pending → decide → occupied
round-trip works end-to-end; the expiry reaper fires `refused` on a timed-out pending request;
`authAdmin` strictly gates `/requests` (no empty-admin fallback — that was the P2 escape hatch for
reclaim/remove, not for governance endpoints). Full suite stays green.

**Experiment** (ADR 069, for batond): does live-approval-by-default measurably raise
time-to-first-`occupied` vs pre-issued grants? The pending→decide round-trip latency + admin
interrupt count are the metrics; the P3.2 audit log emits them.

## Consequences

- `claim` replaces `hello` as the WS session-entry frame. The two coexist on the feature branch until
  the P3.3 atomic cutover removes `hello`/token.
- The pending/push model means admin approval is a push-to-client, not a poll; the WS stays open
  indefinitely (up to the 1h expiry) waiting for a decision — this is the spec-gap 3 contract.
- `authAdmin` with strict `is_admin` (no fallback) is the gate for all P3.2 governance endpoints.
  The empty-admin fallback (P2) stays on `reclaim`/`remove` only.
- Builds on ADR 076 (the grant/request/key substrate), ADR 071 (audit log + admin gate), ADR 070
  (capabilities/account_status), ADR 069 (the plan + four decisions), ADR 061/063 (firehose/observer,
  whose `can_observe` gate wires into the observe-target claim path).
