## Decision

### Wire protocol (SPEC A.3 / ADR 078)

`ClaimFrame` added to `WSClientFrame`. `OccupiedFrame`, `RefusedFrame`, `PendingFrame` added to `WSServerFrame`. All frames exported from `@musterd/protocol` (`packages/protocol/src/frames.ts`).

### Claim handler — 9-step flow (ws.ts)

1. **Version check** — `frame.v !== PROTOCOL_VERSION` → `version_mismatch`
2. **Team lookup** — `requireTeam(ctx.db, frame.team)` → `not_found` if missing
3. **Key verification** — `hashToken(frame.key) === getAgentKeyHash(ctx.db, team.id)` (agent harness) OR `credential_hash` on member row (human credential); refused `forbidden` if neither matches
4. **Target resolution** — `{ seat: name }` → `getMemberByName`; `{ role: name }` → SQL on role column; refused `not_found` for missing seat/role; human credential must match target seat
5. **Account status check** — `resolveAccountStatus(targetMember)`: `disabled`/`banned` → refused with that code
6. **Single-active check** — `hub.connsForMember(targetMember.id).length > 0` → refused `claim_conflict`
7. **Grant path** — if `frame.grant`: `validateGrant` → `consumeGrant` → OCCUPY (step 9); invalid grant → refused `expired_grant` or `forbidden`
8. **No-grant path** — `createRequest(db, teamId, { kind:'claim', from_session:connId, target, surface })` (deduped by triple); `hub.addPending(conn)` with `_claimApproved` callback; send `pending`; `hub.deliverToAdmins(pending)`; audit `claim.pending`; **return** (WS stays open)
9. **OCCUPY** — `attach()` presence; `hub.add(conn)`; `state.authenticated = true`; send `occupied`; emit presence; audit `claim.occupied`

The `_claimApproved(presenceId)` callback captures the WS closure so the HTTP `/decide` path can flip `state.authenticated` from outside without re-opening the socket.

### Target encoding (requests table)

`target` is a single string: `seat:<name>` | `role:<name>` | `observe` (null for teammate-join). Encoded by `encodeTarget(ClaimTarget)` in ws.ts; decoded by prefix-split in the HTTP decide handler.

### HTTP request lane (http.ts, SPEC A.5/A.7)

**`GET /teams/:slug/requests`** — authAdmin-gated; optional `?status=pending`; returns `{ requests: Request[] }`.

**`POST /teams/:slug/requests/:id/decide`** — authAdmin-gated; body = `DecideRequestSchema` (`{ decision: 'approve', lifetime, ttl_hours? }` or `{ decision: 'deny' }`).

Approve path:
1. Resolve target member from encoded target string
2. Account-status check
3. `issueGrant(db, teamId, { scope, target, lifetime }, admin.name)` → gives the session authorization
4. `attach()` presence with `existing.from_session` as connId
5. `decideRequest(db, teamId, id, 'approved', admin.name)` — settle the row
6. `conn._claimApproved(presence.id)` — flip WS auth state
7. `hub.deliverClaimDecision(from_session, occupied_frame)` — push to waiting WS
8. Broadcast presence online; audit `request.decide`; return `{ request_id, decision, delivered }`

Deny path: `decideRequest('denied')` + `hub.deliverClaimDecision(refused_frame)`; audit `request.decide`.

### Hub additions (hub.ts)

On `Connection`: `awaitingClaim?`, `isAdmin?`, `_claimApproved?`.

On `Hub`: `addPending(conn)` (byConn only, not byMember — no deliver routing while pending), `getConn(connId)`, `deliverClaimDecision(connId, frame)`, `deliverToAdmins(teamId, frame)`.

### Reaper extension (reaper.ts)

Before the presence sweep: SELECT pending rows with `expires_at < now`, call `expireRequests(db, now)`, push `refused { code: 'expired_grant' }` via `hub.deliverClaimDecision` for each, audit `request.expired`.

### Audit verbs (audit.ts)

New AuditAction values: `claim.occupied`, `claim.refused`, `claim.pending`, `request.decide`, `request.expired`.

### Approval UI (approvals.tsx)

Login form (team/seat/token) → `LiveConfig`. 10s poll via `fetchRequests(cfg, { pendingOnly: true })`. `onApprove` → `decideRequest(cfg, id, { decision:'approve', lifetime })` (converts UI `GrantLifetime` to protocol form). `onDeny` → `decideRequest(cfg, id, { decision:'deny' })`.

New web client helpers (`client.ts`): `fetchRequests()`, `decideRequest()` — both throw `AuditFetchError` on 401/403 so the route can show admin-specific copy.

## Alternatives

**Client-polls for decision** (vs spec-gap 3 server-push): rejected — the WS stays open anyway for heartbeat; polling adds client complexity and kills the "instant occupation" UX. Server push via the pending WS is cleaner and already in the connection.

**Separate `claims.ts` store** (Cleo's original d08cf43): reverted in favor of June's unified `requests.ts` + `grants.ts` — single schema, single source of truth, no divergent column names.

**Grant validation in decide endpoint**: not done — the POST `/decide` is admin-authoritative; admin decide IS the grant authorization. The issued grant records the approval; the `once`-lifetime grant is consumed immediately on issue.

---

## Consequences

- Claim handshake is fully wired: WS `claim` → 9-step flow → `occupied`/`refused`/`pending`
- Admin request lane is live: `GET /requests`, `POST /requests/:id/decide` (authAdmin-gated)
- Approval UI polls live data and can approve/deny; terminal frame delivered instantly to waiting WS
- No `hello` path removed here — removal is in the same p3-cutover commit
- Security review required before merge: agent-key handling, grant consumption, pending-conn lifecycle

## Observability & Evaluation

**Traces** — every claim outcome audits (`claim.occupied` / `claim.refused` / `claim.pending`) and every admin decision audits (`request.decide` / `request.expired`), all `{ ts, actor, target, result, detail }`, writable to the batond flywheel (ADR 051). Structured log lines `ws_claim_occupied` / `ws_claim_pending` carry team/target/request/conn. The reaper emits `reap_requests_expired` on each tick.

**Eval** — success = the claim handshake is correct, the request lane delivers, and no regression. Bar: full suite stays green (596 tests); the claim handler produces `occupied` for a valid key+grant, `pending` for no-grant, `refused` for invalid key; `/requests/:id/decide` delivers the terminal frame to the waiting WS (asserted by `delivered` in the response). Eval dataset: the integration suite's server tests exercise the new routes via the HTTP handler; the protocol tests cover the union wiring.

**Experiment** — n/a for this wiring ADR. The behavioral experiment (does governed claim with admin approval improve team trust vs ambient key?) is a v0.3 launch eval, not a code-path A/B.
