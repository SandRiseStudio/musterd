# 069 — v0.3 governance: build plan, spec reconciliation, and the four decisions

- Status: accepted — 2026-06-29 (plan); **fully implemented 2026-06-30** — P0/P1 (ADR 070) + P2 (ADR 071) + P3 the hard auth cutover (ADRs 075–078) all landed on `main`, plus the banned-inert read-gate follow-up (PR #37). P4 (credentialed remote join) remains.
- Date: 2026-06-29

## Context

Wave 1 (the coordination-loop hardening) is complete and on main. The next rock is **Wave 2 — v0.3
shared-teams governance**: the enforced layer the wire already anticipates (seats, roles, capabilities,
agent key + grants, the approval lane, audit). It is the prerequisite for the full notification tiers,
schedule enforcement, and safe multi-user/remote teams.

The model is **fully designed** — `SPEC.md` Appendix A (A.0–A.9, normative wire/HTTP), with rationale in
`docs/design/membership-model.md` + `docs/design/security.md` and the two-universes capability split in
ADR 026. SPEC A.0 is explicit that it is **breaking**: the auth unit changes from *per-member token =
identity* to **agent key** (authenticates a harness) **+ admin-issued grant** (authorizes occupying a
seat). The collaboration Envelope/Acts (§2–§3) are **unchanged**; governance is a distinct surface.

Two findings from a build-state audit (2026-06-29) shape the plan:

1. **It is mostly delta, not greenfield.** Single-active + grace (ADR 042/068), the `urgent`/availability
   *mechanism* (ADR 044 — but `can_flag_urgent` is **ungated**: `envelope.ts` validates the reason and a
   comment names the seam), human observers (ADR 063 — the one **enforced** proto-capability: observers
   can't send, `route.ts`), the durable git seat-file roster (ADR 058), and explicit activation all ship
   today. The code even carries `// the v0.3 seat model will gate this` markers at the exact insertion
   points (`route.ts`, `http.ts` reclaim/remove, `envelope.ts` urgent).

2. **A.9's migration text predates ADR 058 and must be reconciled.** A.9 describes migrating a raw
   `members` table (rename → `seats`, drop `token_hash`). But ADR 058 already made the daemon a
   **projection of git `.musterd/seats/*.toml` files** (`seatfile.ts` + `reconcile.ts`); `token_hash`,
   `bound_at`, and `id` are the daemon-private anchors that survive reconcile. So "members → seats" is not
   a fragile data migration — it is **extending the seat-file schema + reconcile projection**, and a daemon
   DB reset is *non-destructive* because the durable roster regenerates from the git files.

Current state (as-built): schema **v7**; `members` has `role` (free text, no capabilities), `kind`,
`lifecycle`, `availability`, `token_hash`, `bound_at`, `observer`; **no** `account_status`, `capabilities`,
`roles`, `grants`, `requests`, or `audit`. `authMember` is `sha256(token) → full member`; **there is no
capability check anywhere** beyond identity-match and observer-mute.

## Problem

v0.3 governance is too large and too breaking for one ADR or one PR. We need a phased decomposition where
each phase ships green, the breaking auth change is isolated to one coordinated moment, and the highest-value
governance (which also completes the Wave-2 notification-tiers item) lands early on the *existing* token auth
without a flag day. We also need to resolve the spec gaps the design left open before writing code.

## Decision

### The four directional decisions (Nick, 2026-06-29)

1. **Migration = one-shot reset.** Bump the schema; write no data-migration code. Safe *because* ADR 058
   made the daemon a projection of the git seat-files — the durable roster regenerates on reconcile and only
   daemon-private state (tokens/credentials) is reminted. SPEC A.9 sanctions a one-shot reset for a pre-1.0
   local product; the dogfood daemon is already empty. (Documented reset; no best-effort migrator.)
2. **Auth rollout = hard cutover to claim/grant everywhere.** No localhost-keeps-token dual-path. Localhost
   and remote both move to *agent key + claim*. Cleaner end state (one auth model); the cost is that the
   cutover (P3) is a single coordinated breaking moment across all surfaces (`init`, `team add`, MCP env,
   `join`). P1/P2 deliberately land **first and non-breaking** so P3 is purely the auth swap.
3. **Durable seat fields live in the git files.** `account_status` (admin-set) + per-seat capability
   narrowing go in `seats/<name>.toml`; role defaults + charter in a git-tracked `roles/<name>.toml`. This
   keeps the ADR 058 property "the durable identity is a file act, git history = the membership/authz audit
   log." **Credentials** (agent-key hash, grant hashes, human-credential hashes) stay **daemon-private**,
   exactly like `token_hash` today — never in the files.
4. **Deliver P0 (this ADR) + start P1**, splitting the single roadmap item into the five phases for review
   and lane-carving with teammates before the breaking P3.

### The five phases (each lands as its own ADR)

- **P0 — this ADR.** Build plan + the four decisions + the spec-gap resolutions below. Doc-only.
- **P1 — Seats data model (ADR 070).** Schema migration (one-shot reset to a v0.3 baseline): `account_status`
  + `capabilities` on the seat; a `roles` concept carrying default capabilities + charter with **per-seat
  narrowing, never widening**. Extend `seatfile.ts` + a new `roles/*.toml` + the `reconcile` projection; lift
  the CLI role-template's already-shaped `capacity`/`charter`/`capabilities` into a shared `@musterd/protocol`
  type. **Permissive defaults, no enforcement yet, token auth unchanged** — pure substrate.
- **P2 — In-band enforcement + audit (ADR 071+).** The first real governance value, and it **completes the
  Wave-2 notification-tiers item**: gate `urgent` on the sender seat's `can_flag_urgent` in `routeEnvelope`
  (+ audit + recipient `wasnt_urgent`); `is_admin` gate on governance routes (replacing the ungated
  reclaim/remove); `visibility_level` viewer-scoped roster projection; `account_status` enforcement
  (non-active not claimable/sendable; `banned` rejects credentials); formalize `can_observe`; append-only
  **audit log** + `GET /audit`. Ships on the **existing token auth** (occupant == seat) — **no flag day**.
- **P3 — Credentials + claim handshake (2–3 ADRs).** The breaking auth rework: team agent key (hashed,
  rotatable) + grants (issue/revoke, lifetime once|ttl|standing) + human credentials; the WS `claim` frame
  replacing `hello`; the no-grant request/approval lane (A.5) with the local-admin fast path and the
  one-keystroke approval card; team `policy` (`allow_pre_issued_grants`); the surface migration (`team add`
  → seat no token; `MUSTERD_TOKEN` → `MUSTERD_AGENT_KEY` + `MUSTERD_CLAIM` (+ optional `MUSTERD_GRANT`);
  `init`/`join` → claim flow). The one coordinated breaking moment (decision 2).
- **P4 — Credentialed remote join (ADR).** Plug the P3 credentials into the already-built ADR 040 secured
  off-loopback bind. Mostly integration + cross-network claim flow + docs.

Roadmap (Wave 3 / Reserved): availability **schedule enforcement** (`off_hours`), `urgent` rate-limiting,
multi-admin delegation, per-seat/rotating keys, signed audit — all named-but-deferred in the design.

### Spec-gap resolutions (normative; the build phases assume these)

The design left four things to the implementation; we pin them here so P1–P3 have no ambiguity:

1. **Credential/grant token format.** Reuse the existing minting scheme (`prefix_ + base64url(randomBytes(24))`,
   stored only as `sha256` hex — `newToken()` today). Distinct prefixes per credential class so a leaked
   secret is self-identifying: `mskey_` (agent key), `msgr_` (grant), `mscr_` (human credential). One shared
   minting/hashing helper; never logged; never in `--json`.
2. **Request expiry.** A pending claim/teammate request expires after a **default 1-hour** window
   (configurable via team policy later), swept by the same reaper that ages presence/grace. On expiry the
   server emits `refused {expired_grant}`-style closure to the waiting claim and marks the request `expired`.
3. **Pending-claim wait contract.** The `claim` WS stays open in `pending`; the server **pushes** the
   terminal frame (`occupied` on approve, `refused` on deny/expire) — no client polling. On a dropped socket
   the client re-`claim`s; the server **dedupes** to the same open request (idempotent by `(team, from_session,
   target)`), so a blip never spawns duplicate admin prompts.
4. **`decide` → grant lifetime binding.** `POST /teams/:slug/requests/:id/decide` takes
   `{ approve: bool, lifetime?: "once"|"ttl"|"standing", ttl_hours?, create_seat?, capabilities? }`. On
   approve the server issues a grant with the chosen lifetime (the membership-model "admin picks the window"
   resolution) and emits `occupied`; `once` → single-use, `ttl` → `expires_at = now + ttl_hours`, `standing`
   → until revoked.
5. **ADR 058 reconciliation (the A.9 drift).** `members → seats` is implemented as: extend the **seat-file**
   schema with `account_status?` (admin-set; `provisioned` is *derived* from never-occupied, not stored) and
   `capabilities?` (narrowing); add `roles/<name>.toml` for role defaults + charter; `reconcile` projects
   both into new daemon columns. Credentials remain daemon-private. The one-shot reset rebuilds the daemon DB
   from the (extended) files — no row-migration code.

## Observability & Evaluation

**Traces** — P0 is a planning ADR and emits no spans itself. The phases it governs each carry their own
Observability section (the obs-evals gate enforces it for 070+). The cross-cutting trace artifact this plan
*introduces* is the **audit log** (P2): every governed operation (`grant issue/use/revoke`, `claim/occupy/
release`, `account-status change`, `key rotation`, `policy change`, `request decision`) writes
`{ ts, actor, action, target, result }` — the coordination-governance trace that no single-agent observability
tool can produce, and a direct feed for the batond flywheel (ADR 051).

**Eval** — the plan's success metric is **sequencing safety**: every phase before P3 lands without breaking an
existing local session (measured by the full test suite staying green on each merge and a dogfood team
surviving each phase un-migrated), and P3 cuts over **all** surfaces atomically (init/team add/MCP env/join) in
one coordinated change with its own migration test. **Dataset/baseline**: the current 502-test green suite is
the regression baseline each phase must hold; the as-built audit (schema v7, zero capability checks) is the
"before." A phase that needs a flag day is a plan failure to be caught at ADR-review, not at merge.

**Experiment** — named for when batond exists: A/B the *governance overhead* of the claim/grant model vs the
v0.2 token model on seeded onboarding + reconnect tasks — does live-approval-by-default measurably raise
time-to-first-`occupy` or admin-interrupt count, and does the TTL-grant lifetime (vs single-use) cut reconnect
re-prompts to ~zero without widening the exposure window? This calibrates the secure-default posture against
its friction cost.

## Consequences

- One reviewable plan instead of a 1500-line breaking PR; teammates can carve non-overlapping lanes (P2
  enforcement + the web/governance UI surfaces are disjoint from the P1 schema work).
- The highest-value governance (capability enforcement + audit, which also finishes notification tiers) lands
  in P2 on the existing token auth — value before the breaking change, no flag day until P3.
- The hard-cutover decision (2) concentrates all breakage into P3; P1/P2 must be written to not depend on the
  new auth, and P3 must migrate every surface in one coordinated ADR set.
- The git-files decision (3) keeps the membership/authz audit trail in git (ADR 058 lineage) and bounds the
  daemon-private surface to credential hashes only.
- The one-shot-reset decision (1) means the v0.3 schema is designed clean (not as a chain of ALTERs); existing
  dogfood daemons reset once, documented, and re-project from their git seat-files.
- Builds on ADR 058 (file roster — the seat seam), ADR 026 (two-universes capability split), ADR 042/063/068
  (single-active, observer, workspace-scoped displacement), ADR 044 (the urgent/availability down-payment this
  governs), ADR 039/040 (the transport P4's credentials ride on).
