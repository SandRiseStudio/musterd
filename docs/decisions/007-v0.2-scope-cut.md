# 007 — v0.2 scope cut: minimal trust model first; defer governance to v0.3

- Status: accepted
- Date: 2026-06-10

## Context

A multi-session design brainstorm grew the planned v0.2 from "fix the auto-join/identity bug" into a full shared-team governance system: seats/roles, a team agent key + admin-issued grants with three lifetime classes, a request/approval lane with approval cards + fingerprints + batching, capabilities with per-seat narrowing, viewer-scoped visibility projections, an audit log, notification tiers, and `urgent`-abuse tracking. Each step was locally reasonable; the **compounding** was not examined until the whole thing was on paper.

## Problem

The machinery is sized for a shared/remote, multi-human team. The actual product today is **local-first, single-user, single-admin, bound to `127.0.0.1`** — its own threat model says the local machine is trusted. Building governance for organizations before user #2 exists is mis-sequenced, and two specifics make it worse:

1. **The agent key + grant model is a security *regression* for the solo case.** The v0.1 per-member token is already least-privilege (acts as exactly one member on one team). A team-wide agent key is broader, which is precisely why grants were then needed to narrow it back down. We introduced a skeleton key and built a locksmith shop to manage it.
2. **Live-approval-by-default deadlocks the solo operator.** The person interrupted to approve the agent is the same person who just launched it; and the server cannot actually verify the "admin co-present" fast path without more machinery (a harness can assert it falsely). See pressure-test breaks 1–2 below.

Meanwhile this is a pre-launch OSS project whose wedge is a 90-second demo, a crisp README, and a reserved npm name — none of which need approval cards.

## Decision

**Ship a minimal v0.2 that fixes only the real bug, and defer the governance system to v0.3 (shared teams), kept fully designed on paper so nothing is lost.**

**v0.2 — minimal trust model (build now):**
- **Explicit activation.** The MCP adapter is dormant by default (tools available, no auto-join); a `team_join` tool activates; opt-in `MUSTERD_AUTOJOIN` env keeps the one-keystroke magic. *(fixes consent)*
- **Single-active + refuse + 45s grace**, server-enforced per member. A second live presence for a member is refused, not silently shadowed; reconnect within grace keeps the seat. *(fixes "3 sessions = 3 minds wearing one name")*
- **Keep v0.1 per-member tokens.** No agent key, no grants, no request/approval lane, no audit table.
- **`working: x` activity** with persist-while-alive + freshness staleness (the two-clocks rule) — it's cheap and it's what makes the roster feel alive in the demo.
- Optional, only if trivial: `disabled`/`archived` account states (one column + two CLI verbs). Defer `provisioned`/`banned` nuance.
- Protocol bump to **`musterd/0.2`** (small: single-active behavior + activity fields), not the big one.

**v0.3 — shared-teams governance (designed, not built):** seats/roles, agent key + grants + approval lane, capabilities + visibility projection, audit, notification tiers + `urgent`, human observers. This is the `membership-model.md` / `spec-v0.3-draft.md` / `security.md` set. It activates when the daemon stops being localhost-only — the moment its threat model becomes real.

## What survived the pressure test (kept, for the record)

`single-active + grace` (clean under every scenario), explicit activation, `working`/staleness two-clocks, human-only observers, charter-as-data, memory-as-reserved-seam, and the three-axis state *taxonomy* (we build a slice now). These are real and carried into v0.2 or v0.3 unchanged.

Landmines the brainstorm pre-answered (now documented in the v0.3 set): grant expiry must gate *new* claims and never evict a live one (break 3); fingerprints are a *recognition* aid, not a security control (break 5); `urgent` strike-tracking needs ≥2 humans to matter (break 6).

## The one break that survives into v0.2 and is NOT governance

**Dead-air (break 4).** MCP is pull-only — the server cannot push to the model. A dormant-then-joined agent only sees team messages when *it* calls `team_inbox_check`; an agent that never checks is a teammate who never reads Slack. This is a real UX cliff at our layer, independent of all the governance. Mitigations to design during v0.2 build: the `team_join` tool result + charter strongly instruct inbox checks at task boundaries; document a harness-hook pattern (e.g. Claude Code hooks injecting a reminder). Tracked as the open question for Milestone 3.

## Consequences

- `membership-model.md`, `security.md`, and the renamed `spec-v0.3-draft.md` are re-tagged **v0.3 target**; `membership-impl-plan.md` is re-scoped to the ~3-milestone minimal v0.2.
- The breaking governance protocol change (seats/grants) moves from "ADR 007" (this) to a **future ADR** written when v0.3 is built.
- Net: ship the small fix, record the demo, publish; let real users (shared teams) pull the governance work forward when it earns its keep.
