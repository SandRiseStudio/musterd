# 076 — v0.3 P3.1: credential + grant substrate

- Status: accepted — 2026-06-29 (P3.1 of ADR 069; P3 lane taken over from Jasmine, see below)
- Date: 2026-06-29

## Context

ADR 069 phased v0.3 governance; P0–P2 (ADRs 070/071) landed the seat/capability substrate + in-band
enforcement + audit on the **existing token auth** (occupant == seat, no flag day). P3 is the breaking
phase ADR 069 isolated: agent keys, grants, human credentials, the WS `claim` handshake that replaces
`hello`, and the request/approval lane — the move from "hold a seat token" to "present a credential and
claim a seat."

no localhost token dual-path, ever), and — with Jasmine (the P3.1/P3.2 owner) unavailable — **take over

This ADR is **P3.1: the durable substrate** — the tables, the credential/grant minting, and the admin
write APIs that the handshake (077) and the CLI (075) build on. It is **additive and non-breaking on its
own** (new tables, new admin-gated endpoints; `hello`/token auth still works). The breaking removal of
`hello`/token is P3.3's atomic cutover — but per Nick's gate this substrate is **built on a branch against
an in-memory/dev daemon and lands in the single coordinated P3 cutover merge**, not incrementally on the
live daemon.

## Problem

Stand up the credential/grant/request storage + admin operations the rest of P3 needs, faithful to the
normative SPEC (A.2 credentials, A.5 requests, A.7 HTTP deltas, A.9 migration) and the four ADR 069
implementation decisions, **without** prematurely entangling the breaking parts (claim frame handling,
hello removal) that belong to 077/075. Specifically:

1. **Where secrets live.** Per-seat tokens live on `members.token_hash` today, but the **agent key** is
   _team-scoped_ (one per team, rotatable) and **policy** is _team-scoped_ — neither has a home on a
   member row. Human **credentials** are per-human (per member). Grants and requests are their own rows.
2. **Minting + storage discipline.** SPEC A.2: servers MUST store **only hashes**; a credential/key/grant
   secret is shown **once** at mint and never logged. Reuse the proven `prefix_ + base64url(randomBytes)`
   - sha256-hex scheme, with the P3 prefixes.
3. **Admin write surface.** The grant/key/policy operations are new admin-gated HTTP endpoints; they must
   match the existing route + error-envelope + audit conventions and use the **strict** `is_admin` gate
   (no empty-admin fallback — these are v0.3-only endpoints with no prior open behavior to preserve).

## Decision

### 1. Storage — migration v10 (one-shot reset, ADR 069 decision 1)

v10 adds the substrate. Team-scoped secrets go on `teams`; grants + requests are new tables:

```
ALTER TABLE teams ADD COLUMN agent_key_hash TEXT;          -- one rotatable team key; null until set
ALTER TABLE teams ADD COLUMN policy TEXT;                  -- JSON: { allow_pre_issued_grants: bool }
ALTER TABLE members ADD COLUMN credential_hash TEXT;       -- per-human credential; null for agents/seats

grants(id PK, team_id → teams ON DELETE CASCADE, scope, target, token_hash, issued_by,
       lifetime, expires_at, single_use, revoked, created_at)
requests(id PK, team_id → teams ON DELETE CASCADE, kind, from_session, target, status,
         decided_by, created_at)
INDEX idx_grants_team   ON grants(team_id)
INDEX idx_requests_team ON requests(team_id, created_at)
```

- `scope` ∈ `seat|role`, `target` = the seat/role name; `lifetime` ∈ `once|ttl|standing`; `expires_at`
  nullable (set for `ttl`); `single_use` bool (the `once` consumption flag, consumed like `members.bound_at`).
- `kind` ∈ `claim|teammate`; `status` ∈ `pending|approved|denied|expired`. Stored as open `TEXT` (no
  `CHECK`) so widening the enum later needs no table rebuild (the v5 CHECK-rebuild trap).
- Migration body follows the v8/v9 pattern (`db.exec` DDL; the runner stamps `schema_version` — `up()`
  must not). `schemaVersion()` → 10 surfaces in `/health`.

### 2. Minting + hashing (ADR 069 decision 1)

Parameterize the existing `newToken` to `newToken(prefix = 'mskd_')` and add named minters:
`newAgentKey()` → `mskey_…`, `newGrantToken()` → `msgr_…`, `newCredential()` → `mscr_…`. Storage is
always `hashToken(secret)` (sha256-hex, reused). The plaintext is returned exactly once from the mint
call and never persisted or logged. Validation mirrors `authMember`: hash the presented secret, look up by
hash, throw `MusterdError` on miss; `single_use`/`expires_at`/`revoked` are checked at validation and a
`once` grant is consumed on first successful use.

### 3. Stores — `store/grants.ts`, `store/requests.ts`

- `grants.ts`: `issueGrant`, `revokeGrant`, `getGrantByHash` (validation: not revoked, not expired,
  consume `single_use`), `listGrants`.
- `requests.ts`: `createRequest` (**idempotent dedup by `(team, from_session, target)`**, ADR 069
  decision 3 — re-create returns the open request), `decideRequest` (status + `decided_by`), `listRequests`,
  `expireRequests` (the 1h-default reaper, ADR 069 decision 2 — wired by 077). Team-secret writers
  (`setAgentKeyHash`, `rotateAgentKey`, `setPolicy`, `setCredentialHash`) live next to the member writers.

### 4. Admin HTTP endpoints (SPEC A.7), strict `authAdmin`

Added as new `rest` branches in `http.ts`, each `authAdmin`-gated, Zod-validated body, audited:

| Method + path                        | Body                                                                               | Audit verb      |
| ------------------------------------ | ---------------------------------------------------------------------------------- | --------------- |
| `POST /teams/:slug/grants`           | `{ scope, target, lifetime, ttl_hours?, single_use? }` → `{ grant: mskgr_… once }` | `grant.issue`   |
| `DELETE /teams/:slug/grants/:id`     | —                                                                                  | `grant.revoke`  |
| `POST /teams/:slug/agent-key/rotate` | — → `{ agent_key: mskey_… once }`                                                  | `key.rotate`    |
| `POST /teams/:slug/policy`           | `{ allow_pre_issued_grants }`                                                      | `policy.change` |

`GET /requests` + `POST /requests/:id/decide` are **claimed by 077** (they belong with the request-lane +
decide→grant binding); this ADR provides the `requests.ts` store they call. The `account_status.change`
write API (ADR 071 noted P2 omits it) also lands here as `POST /members/:name/account-status` (audit
`account_status.change`), since it reuses `setMemberGovernance`.

### 5. Protocol schemas (coordinated with 078)

New `@musterd/protocol` modules: `grants.ts` (`GrantSchema`, `LifetimeSchema`), `requests.ts`
(`RequestSchema`, `RequestStatusSchema`), `policy.ts` (`PolicySchema`), `credentials.ts` (mint-response
shapes), barrelled in `index.ts`. The new error code `expired_grant` (SPEC A.8) is mine; `claim_conflict`
is riley's (078) — we append to `errors.ts` in non-overlapping hunks. riley's WS frames (078) **import**
these types so the handshake and the substrate share one contract.

## Observability & Evaluation

**Traces** — every admin governance op audits (extends ADR 071): `grant.issue` / `grant.revoke` /
`key.rotate` / `policy.change` / `account_status.change`, each `{ ts, actor, target, result, detail }`,
with secrets **never** in `detail`. A denied admin op (non-admin caller) audits `result: deny` so the
existing audit-log view (ADR 073) + CLI reader (ADR 074) surface the new verbs with no change (their
`action` is an open string). Grant _consumption_ (a `once` grant spent, a grant validation refused as
`expired`) is audited too, so the substrate's own lifecycle is observable.

**Eval** — success = **the substrate is correct + additive (no regression) before any breaking cutover**.
Bar: the full suite stays green; with no agent key / no grants / no policy set, the existing token-auth
team behaves exactly as P2 (this substrate changes nothing until 077/075 wire it). New assertions, all on
the in-memory daemon harness: a minted secret validates by hash and never appears in storage or logs;
a `once` grant validates once then refuses; a `ttl` grant refuses past `expires_at`; a `standing` grant
persists until revoked; a non-admin is refused every new endpoint (`deny` audited); request creation
dedupes by `(team, from_session, target)`; migration v10 is reversible-by-reset and round-trips the new
columns. **Baseline/dataset**: the green P2 suite + the v9 schema are the "before"; each minter/endpoint
ships with its paired allow/deny + lifecycle test as the regression guard.

**Experiment** — for batond (ADR 069): once 077/075 land, the grant lifetime distribution (`once` vs `ttl`
vs `standing`) and request approve/deny/expire rates become coordination signals — e.g. does the 1h
request-expiry default (decision 2) cause real claim drop-offs in onboarding, and does `allow_pre_issued_grants`
materially cut approval latency? The grants + requests tables are the measurement substrate.

## Consequences

- The breaking auth rework gets a **clean, additive, independently-testable foundation**: P3.1 can be fully
  built + verified (minting, storage discipline, admin ops, audit) before the cutover wires it, shrinking
  the risk concentrated in the single P3 merge.
- Team-scoped secrets (`agent_key_hash`, `policy`) on `teams` + per-human `credential_hash` on `members`
  matches SPEC A.9 and keeps the daemon-private secrets off the git seat-files (ADR 069 decision 3).
- The substrate **deliberately stops short** of claim-frame handling and `hello` removal (077/075), so this
  ADR is non-breaking and could in principle ship early — but per Nick's gate it lands in the one atomic P3
  cutover merge, built on a dev daemon, with a security review (credentials + breaking auth) before merge.
- Open coordination item for the team: P3.1/P3.2/P3.3 must **converge into one merge** — a shared
  `p3-cutover` integration branch is the likely mechanism (each lane's branch merged in, hello/token removed
  in the same commit, live daemon reset once). Raised separately for a team decision.
