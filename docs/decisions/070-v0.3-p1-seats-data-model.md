## Context

ADR 069 phased the v0.3 governance rock. **P1 is the substrate**: the seat/role/capability data model
that P2 enforcement and P3 credentials both build on. P1 ships **no enforcement and no auth change** â€”
it adds the *shape* (capabilities on a seat, roles carrying defaults, account status) with
backward-compatible permissive defaults, so the test suite stays green and no dogfood session breaks.

Per ADR 069 decision 3, the durable seat fields live in the **git seat-files** (ADR 058 lineage);
credentials stay daemon-private. Per ADR 026, only **Universe-1** capabilities are musterd-enforced
(later, in P2); **Universe-2** (`tool_allowlist`, `declared_resource_scopes`) are declared/provisioned
only â€” musterd stores and serves them, never enforces.

Today (schema v7): a member's `role` is free text carrying nothing; there is no `account_status` or
`capabilities`. The CLI role-template (`onboard/role.ts`) already carries a richer
`capacity`/`charter`/`tools` shape that **never reaches the server** â€” "the most fully-shaped reserved
structure awaiting the v0.3 wire type." P1 lifts the identity half of that into `@musterd/protocol`.

## Decision

### The capability shape (the v0.2 fixed set â€” no RBAC engine)

A single `Capabilities` record in `@musterd/protocol`, the seven fields the design enumerates
identically in SPEC A.1 / security.md / membership-model.md, split by universe:

**Universe-1 â€” musterd-enforced (in P2):**
- `is_admin: boolean` â€” governance authority (governance routes, see-all visibility).
- `can_flag_urgent: boolean` â€” may set `meta.urgent`.
- `can_observe: boolean` â€” may attach read-only as an observer (humans only).
- `can_message: "team" | "none"` â€” may send acts to the team (`none` = muted). (The "specific roles"
  scope the design mentions is a roadmap refinement; v0.3 ships `team | none`.)
- `visibility_level: "admin" | "team"` â€” `admin` sees everything (credentials/grants/audit/policy/all
  charters); `team` is the need-to-know projection (teammate handles + presence + acts addressed to it).

**Universe-2 â€” declared/provisioned only (never enforced by musterd, ADR 026):**
- `tool_allowlist: string[]` â€” declared tool/MCP scopes.
- `declared_resource_scopes: string[]` â€” declared repo/dir scopes.

### Defaults â€” backward-compatible (current behaviour = everyone may do everything)

The **generalist default** (no role, no per-seat override) preserves today's behaviour exactly:
`{ is_admin:false, can_flag_urgent:true, can_observe:true, can_message:"team", visibility_level:"team",
tool_allowlist:[], declared_resource_scopes:[] }`. `can_flag_urgent:true` keeps urgent ungated until P2
flips the default; `visibility_level:"team"` matches today's roster (no credentials are exposed today,
so the projection is a no-op until P2 adds the things to hide). The **team creator's human seat**
defaults `is_admin:true` (admin = creator default).

### Roles carry defaults; seats narrow, never widen

A **Role** is `{ name, capabilities (partial defaults), charter? }`, stored as a git
`roles/<name>.toml`. A seat in a role inherits the role's defaults; a seat-level `capabilities` block
**narrows only** â€” a boolean may go `trueâ†’false` but not `falseâ†’true`; a scope may shrink
(`teamâ†’none`, `adminâ†’team`) but not widen; a list may subset. The **effective** capabilities are
`clampNarrow(generalist âŠ• roleDefaults, seatOverride)`; the projection clamps any attempted widening
back to the role ceiling (in P1 it clamps silently; P2's admin endpoints reject a widening write).

### Account status

`account_status: "provisioned" | "active" | "disabled" | "banned" | "archived"` (SPEC A.6 state
machine). The **admin-set** status (`disabled`/`banned`/`archived`) lives in the seat-file; `active` vs
`provisioned` is **derived** from occupancy (never-occupied â‡’ `provisioned`, ADR 058 `bound_at` is the
"has been held" anchor) when no admin override is set. So the file stores intent (the durable,
git-auditable admin decision), and the daemon resolves the live value. Non-`active`/`provisioned`
states gate claim/send in P2; in P1 they are stored + projected only.

### Where it lives (ADR 069 decision 3)

- `seats/<name>.toml` gains optional `account_status?` (admin override) and `capabilities?` (narrowing).
- New `roles/<name>.toml`: `capabilities` (partial defaults) + `charter?`.
- `reconcile` projects role-defaults âŠ• seat-narrowing â†’ an effective-capabilities blob + resolved
  `account_status` onto new daemon columns; the public `Member` exposes them for the roster.
- Migration is a **one-shot reset** to a v0.3 baseline schema (ADR 069 decision 1) â€” the daemon
  rebuilds from the extended files; no row-migration code.

### What P1 does NOT do

No enforcement (P2), no credentials/claim (P3), no admin HTTP endpoints (P2/P3), no `can_message`/
`visibility` projection behaviour change (P2). P1 is the typed substrate + its projection + tests.

## Observability & Evaluation

**Traces** â€” P1 adds no acts and no spans; it is a data-model layer. The downstream trace effect is that
the public `Member` (already on every roster read and every `presence`/`deliver` projection) now carries
`account_status` + effective `capabilities`, so when P2 starts emitting governance/enforcement spans they
attribute against a seat whose authority is already legible. No telemetry is emitted by P1 itself.

**Eval** â€” success = **pure substrate, zero behaviour change**: the full suite stays green, the generalist
default reproduces today's roster byte-for-byte (a golden test: a team with no roles/overrides projects
the same `Member` capability-less consumers saw before, plus permissive caps), and `clampNarrow` provably
never widens (property test: for random role/seat pairs, every effective field â‰¤ the role ceiling).
**Dataset/baseline**: the v7 roster output is the "before"; the regression baseline is the green suite.
A behaviour change in P1 is a defect (it belongs in P2).

**Experiment** â€” none for P1 (no enforced behaviour to A/B). The capability shape itself is the variable
the later batond experiment (ADR 069) varies; P1 just fixes the schema it varies over.

## Consequences

- The capability shape is pinned once, in the protocol package, shared by CLI + server + (later) the MCP
  adapter â€” no divergent definitions.
- Permissive defaults + derived `account_status` mean P1 merges with no flag day and no migration code
  (one-shot reset only), exactly as ADR 069 sequenced.
- The narrowing-only rule is enforced structurally (`clampNarrow`) from day one, so P2's admin endpoints
  inherit a correct projection rather than re-deriving it.
- Universe-2 fields are carried as declared data, honouring ADR 026 (we store/serve, never enforce).
- Builds on ADR 069 (the plan), ADR 058 (seat-files + reconcile), ADR 026 (two universes), ADR 044/063
  (the urgent + observer proto-capabilities this generalizes).
