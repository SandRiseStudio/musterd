# Roadmap

musterd v0.1 is deliberately small: a protocol (`SPEC.md`), a local team server, a human CLI, and a universal MCP adapter. The schema and wire format already **reserve** the fields the items below need, so adding them does not break v0.1 clients. Nothing here is required for v0.1 conformance.

## Reserved in v0.1, built later

- **The human↔agent collaboration loop.** *(Near-term — the thesis-central category, evidence-backed; see `docs/design/research-foundation.md`.)* musterd has built the half that lets agents *be* members (identity, presence, collision handling); the measured payoff is the half that wires the loop on the **human's** side. (1) **Driver co-presence** — when a human steers an agent inside its session, the roster must show the human *present*, not offline (the founding dogfood wound; dynamics §1/§54). **Pulled to pre-launch** (2026-06-17) because the flagship demo's headline is "humans and agents as peers." (2) **Reachability / notification** — a human should be *notified* when an agent `request_help`s, not only when watching `inbox --watch`. Co-Gym's ablation shows the notification protocol more than doubles collaboration win rate (30%→70%); a minimal nudge is a pre-1.0 fast-follow, the full **notification tiers** are the v0.3 governance set. (3) **A terminal "done" signal** for a thread (`accept` ≠ finished) — a new act / SPEC bump that serves both progress-awareness and the board layer below.
- **Telemetry & observability.** ✅ **Layer 1 server instrumentation shipped** (ADR 015): one span per Envelope on the validate→persist→route path + act/team metrics, off by default, OTLP-out, no phone-home; the server records the `meta.otel` trace-context convention (ADR 011, now accepted) so a handoff links the sender's and receiver's traces across runtimes and vendors, and `@musterd/mcp` emits/honors `meta.otel`. The §4 metric set is complete (envelope counter/latency/errors/presence-churn + the `presence.active` / `inbox.lag` observable gauges). Next: a full CLI/MCP telemetry SDK, then Layer 2. Strategy, build/buy line, and the **coordination observability** product thesis (MAST-aware views over the act-typed log — the layer existing agent-observability tools can't see): `docs/design/observability.md`; standalone-product branding brainstorm: `docs/design/brand-coordination-observability.md`.
- **Schedule & lifecycle enforcement.** `members.availability` and `lifecycle: until` are stored today but not enforced at runtime. Later: honor availability windows for routing/notifications and auto-expire `until` members.
- **Step-level streaming transport.** v0.1 sends whole Envelopes. A v2 transport option adds step-level streaming (the StreamMA finding: step-level streaming beats wait-for-complete for collaborating agents). The `broadcast` recipient kind is already distinct on the wire to anticipate richer delivery semantics.
- **Cross-network teams (deployment topology & remote transport).** The identity/presence/transport split is already topology-agnostic and the durable inbox tolerates partitions, so members on different machines/networks are *conceptually* a valid team today — but the networking substrate isn't designed: members all dial one daemon that defaults to `127.0.0.1`, and remote transport security is only gestured at. Near-term answer is docs-only (stand on a Tailscale/WireGuard-style overlay — Topology B); then a secured off-loopback bind (TLS-required, `wss://`, paired with the v0.3 agent-key+grant credentials); a hosted relay is the frictionless future. Still one team = one daemon (this is *not* federation). Design: `docs/design/deployment-topology.md`.
- **Team-to-team federation.** A Member belongs to one Team in v0.1 (see ADR 001). Federation — Teams that can address one another, and identities recognized across Teams — comes later; `broadcast` is reserved partly for this.
- **More Surfaces.** v0.1 implements `cli`, `claude-code`, `codex`. The `Surface` enum already includes `web`, `ios`, `slack`, `other`. Planned: a **web dashboard** (designed now in `docs/design/figma-brief-dashboard.md`, built later), an iOS app, and a Slack surface — same Member, more Presences.
- **Work items, board view & the insight layer.** Threads are already proto-work-items (the opening act creates the item; the acts within are its transitions). Later: a kanban-style board and team analytics (time-to-unblock, cycle time, load distribution, bottlenecks) **derived as views over the message log — never stored beside it** — plus a declared *backlog* noun for planned work (the one declaration that can't rot, because it creates intent rather than mirroring execution). Design + cautions (Goodhart, human-vs-agent measurement asymmetry): `docs/design/human-agent-dynamics.md` §4; open brainstorm agenda (planning noun, leadership reporting at altitudes, flow + cost-per-item metrics, the human "waiting-on" bottleneck view): `docs/design/planning-and-insights-brainstorm.md`. Natural home: the web dashboard. The same derived views are the seed of the standalone coordination-observability product (`docs/design/observability.md` §5).
- **Harness tool environment & musterd's own harness.** Today `musterd init` writes one thing into a harness: its own MCP-server entry. Later, a **Role** becomes a harness-agnostic *provisioning template* (charter + capability defaults + MCP/tool recipe), rendered per-harness by the adapter — the **two universes** split (in-band acts musterd enforces vs harness tools it *provisions + declares*; ADR 026). Phased: support all existing harnesses → provision the full per-harness environment → **musterd's own harness** → **mixed-harness teams** (humans/agents on one team across different harnesses, including musterd's own). Provisioning is a starting point, **not a security boundary**, and stays under the non-invasive-coexistence guardrail (additive / reversible / non-obligating; ADR 027) and *compose, don't capture* (defer to MCP / git / the harness; ADR 028). Design: `docs/design/provisioning-recipe.md` (the consolidated recipe + local onboarding flow), ADRs 026–030.
  - **Status:** Phase 1 shipped renderers for **all three required harnesses — Claude Code, Cursor, and Codex** (role MCP servers; Claude Code also provisions permission defaults; Codex writes the project-local `.codex/config.toml` via a minimal `[mcp_servers.*]`-scoped TOML helper — no TOML dep — resolving the global-scope tension by writing project-local, ADR 031), plus the role-template format + built-in library, `musterd role` (list/show/create), the uninstall manifest, charter injection, and `musterd uninstall` (ADRs 029–031). **Phase 2 (local claim-on-first-use) shipped** — folder claim policy (`MUSTERD_CLAIM`), overloaded `team_join`, `musterd claim`, client-side pending presence, all on existing primitives with **no wire change** (ADRs 032–033).
  - **Follow-up (deferred, deliberate — ADR 033):** `musterd claim` delivers identity via the workspace binding (next launch / CLI), **not** live IPC into an already-running unclaimed adapter; in-session live claim is `team_join`'s job. To make `musterd claim --for <code>` *also* bring a running pending session online, an adapter would need to **watch its `.musterd/pending/<code>` marker for a resolution and auto-join** (a small, well-guarded poll). Left unbuilt to keep the mechanism testable without cross-process timing; the marker schema + write/clear path already exist, so this is an additive enhancement, not a redesign.
  - **Follow-up (unblocked by Phase 2):** the **free-text role label vs. role template** unification (`provisioning-recipe.md` "open / fast-follow") — claim now assigns the seat's role, so a template pick can drive the label next. Not yet wired.
- **Sandboxed runtime.** musterd connects agents; it does not run them. A later, optional sandboxed runtime could host members that have nowhere else to live.
- **Python client SDK.** A fast follow after launch. The protocol is language-neutral by design; the TypeScript `@musterd/protocol` is the reference, not the only possible implementation.

## Explicitly out of scope (by principle, not timing)

- **A planner / orchestrator / task-decomposer role.** Principle 5: one member does the work; the team does the coordination. musterd never forces decomposition. A team of one agent (plus optionally a human) is a first-class, even default, configuration.
- **Running your agent.** Principle 4: protocol over framework. We connect agents; we don't own their execution loop.

## How priorities are decided

The wedge is **persistent teams with identity, presence, and humans as peers** — the coordination layer where, per MAST (arXiv 2503.13657), ~79% of multi-agent failures actually happen. Roadmap work is weighed by whether it strengthens that layer (identity, durable inboxes, typed collaboration acts, human partnership) rather than by adding more agents or more orchestration. Why that layer is defensible — neighboring frameworks (e.g. Flue) build deep per-agent durability and delegate coordination to the substrate, stopping exactly where musterd starts — is documented in `docs/design/landscape.md`.

**Human partnership ranks first among the strengtheners, on evidence.** Co-Gym (arXiv:2412.15701; `docs/design/research-foundation.md`) shows collaborative agents beat fully autonomous ones on real-user preference (86/74/66%), that dual control is real (humans co-act 21–32% of actions), and — in its sharpest result — that removing the notification protocol *more than halves* the win rate. So the human↔agent loop above (co-presence, reachability/notification, the done signal) is weighed above further agent-side plumbing: musterd has under-invested there, and it is where the measured value is.

Changes to `SPEC.md` are versioned and gated by an ADR (`docs/decisions/`). New collaboration **acts** or envelope-required fields are a spec-version bump.
